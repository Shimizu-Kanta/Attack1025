# アタック1025 システム概要

## 1. 目的

ポケモン全1025匹を題材に、アタック25風の盤面ゲームとポケモン収集・探索要素を組み合わせたゲームを作成する。

プレイヤーは、盤面上にランダム配置されたポケモンを、ゲーム・カード・グッズ・現地写真などの手段で新たに「捕まえる」ことでパネルを取得する。  
最終的に、より多くのパネルを獲得したチームの勝利とする。

---

## 2. ゲーム概要

本ゲームは、ポケモンの図鑑番号1〜1025を盤面上のパネルとして使用する、探索型の陣取りゲームである。

基本盤面は32×32マスの1024マスとし、必要に応じてワイルド枠1マスを加えることで、全1025匹を扱える構成とする。

ただし、初期実装では5×5、8×8、16×16、32×32など、任意の盤面サイズを選択できるようにする。

### 基本方針

- 盤面にはポケモンがランダムに配置される
- プレイヤーには最初から全ポケモンの位置は公開されない
- 公開されたポケモンに対して、プレイヤーは取得申請を行う
- GMが申請を承認すると、そのチームが該当パネルを取得する
- 取得したパネルの周囲8マスが公開される
- 取得したパネルによって自分の色で他チームのパネルを挟んだ場合、挟まれたパネルは自分の色に変わる
- 最終的に最も多くのパネルを所有しているチームが勝利する

---

## 3. プレイヤー・チーム構成

### 参加人数

- プレイヤーは2〜8チームを想定
- 1チームあたり1〜4人程度
- チームごとの人数は一致していなくてもよい
- GMなしでも進行できる形を将来的な目標とするが、初期実装ではGMありを前提とする

### チームカラー

最大8チームを想定し、ポケモンのバージョンカラーに合わせて以下の色を使用する。

| チームカラー | HEX | 備考 |
|---|---|---|
| 赤 | `#E53935` | 初代「赤」イメージ |
| 青 | `#1E88E5` | 初代「青」イメージ |
| 緑 | `#43A047` | 初代「緑」イメージ |
| 黄色 | `#FDD835` | ピカチュウ・黄色イメージ |
| 金 | `#D4A017` | 金銀の「金」イメージ |
| 銀 | `#9E9E9E` | 金銀の「銀」イメージ |
| ヴァイオレット | `#6A1B9A` | バイオレット版イメージ |
| スカーレット | `#D81B60` | スカーレット版イメージ |

色だけで判別しづらい場合に備え、将来的には各チームにアイコンを設定できるようにする。  
盤面上では、チームカラーとアイコンを組み合わせて表示することで視認性を高める。

## 4. 盤面設定

### 盤面サイズ

ゲーム開始時に盤面サイズを選択する。

初期実装では以下を想定する。

- 5×5
- 8×8
- 16×16
- 32×32

32×32の場合は1024マスとなり、ワイルド枠1マスを加えることで1025匹に対応できる。

### 使用するポケモン番号

盤面に配置するポケモンは、図鑑番号1〜1025の中から選択する。

ゲーム開始時に、使用する番号範囲や除外番号を設定できるようにする。

#### 設定例

```txt
使用する図鑑番号：1〜1025
除外する図鑑番号：25, 150, 493
```

#### 想定する設定項目

- 使用開始番号
- 使用終了番号
- 除外番号
- 将来的には世代別・カテゴリ別のフィルターも検討する

### 注意事項

盤面のマス数より使用可能なポケモン数が少ない場合、盤面を生成できない。

例：

```txt
8×8 = 64マス
使用可能ポケモン数 = 50匹
→ 盤面生成不可
```

この場合は、盤面サイズを小さくするか、使用範囲を広げる必要がある。

## 5. ランダムシード

盤面生成時にはランダムシードを使用する。

ランダムシードとは、同じ条件で同じ盤面を再現するための文字列である。

### 例

```txt
attack1025-20260512-143022-k8d3pa
```

同じシード、同じ盤面サイズ、同じ使用番号範囲、同じ除外番号で盤面を生成した場合、同じポケモン配置を再現できる。

### 保存するべき情報

- ランダムシード
- 盤面サイズ
- 使用する図鑑番号範囲
- 除外番号
- 使用したポケモン一覧データのバージョン

これにより、バグ検証や公平性確認がしやすくなる。

## 6. パネル取得ルール

### 取得可能なパネル

プレイヤーが取得申請できるパネルは、原則として以下の条件を満たすものとする。

- すでに公開されている
- まだ取得されていない
- すでに取得されているパネルの周囲8マスに存在する

ゲーム開始時点では、初期パネルまたは初期公開範囲を設定して、そこから盤面を広げていく。

### アタック25との違い

本ゲームでは、アタック25のように「挟める場合は必ず挟まなければならない」という制約は設けない。

プレイヤーは取得可能なパネルの中から、自由に狙うパネルを選択できる。

### 挟み反転ルール

取得したパネルによって、縦・横・斜め方向に自分のチーム色で他チームのパネルを挟んだ場合、挟まれたパネルはすべて自分の色に変化する。

#### 例

```txt
赤 - 青 - 青 - 赤
```

この状態になった場合、間の青パネルは赤チームのパネルになる。

反転判定は以下の8方向に対して行う。

- 上
- 下
- 左
- 右
- 左上
- 右上
- 左下
- 右下

## 7. 取得申請ルール

### 基本方針

取得証拠となる画像は、原則としてDiscordなどの外部チャットに投稿する。  
ただし、パネル取得に関する正式な申請、先着判定、GM承認、盤面反映、ペナルティ処理はゲームシステム上で行う。

現時点では、証拠画像の投稿先としてDiscordを推奨する。  
Discordは投稿URLを取得しやすく、チャンネル分けや将来的なBot連携にも向いているためである。

### 基本フロー

プレイヤーは、以下の順番で取得申請を行う。

```txt
1. 対象ポケモンの証拠画像をDiscordに投稿する
2. ゲームシステム上で対象パネルを選択する
3. 申請フォームに必要情報を入力する
4. Discord投稿URL、またはGMが確認できる情報を添付する
5. システム上で取得申請を送信する
6. GMが申請内容を確認し、承認または却下する
```

### 先着判定

パネル取得の先着判定は、Discordへの画像投稿時刻ではなく、ゲームシステム上の申請時刻を基準とする。

同じパネルに対して複数チームから申請があった場合、システム上で最も早く申請されたものを優先してGMが確認する。

Discord上で先に画像を投稿していた場合でも、システム申請が遅れた場合は、先にシステム申請を行ったチームが優先される。

そのため、プレイヤーは画像投稿後、速やかにシステム申請を行う必要がある。

### 申請に含める情報

システム上の申請には以下の情報を含める。

- チーム
- プレイヤー名
- 対象パネル
- 対象ポケモン
- Discord投稿URL、またはGMが確認できる証拠情報
- コメント

### Discordの役割

Discordは、以下の用途で使用する。

- 証拠画像の投稿
- 参加者全員への取得状況の共有
- ゲーム中の盛り上がり作り
- GMによる証拠確認
- 将来的なBot連携の土台

### システムの役割

ゲームシステムは、以下の用途で使用する。

- 正式な取得申請
- 申請時刻の記録
- 先着判定
- GM承認・却下
- ペナルティ処理
- パネル取得処理
- 周囲公開処理
- 挟み反転処理
- スコア更新
- ログ管理

### 取得証明として認めるもの

プレイヤーは、指定されたポケモンを何らかの手段で新たに取得・発見し、その証拠をDiscordに投稿する。

想定する取得手段は以下の通り。

- ゲーム内での捕獲・入手
- ポケモンカード
- グッズ
- 現地写真

### 現地写真の条件

写真は、購入できないもの、またはその場にしかないものを対象とする。

例：

- ポケモンマンホール
- ポケモンセンターの入口
- ポケパーク等に設置されているポケモン
- イベント会場の展示物

### 対象外

以下は原則として対象外とする。

- これまでに捕まえていたポケモン
- これまでに撮影していた写真
- 過去に所有していたグッズの使い回し
- インターネット上の画像
- 他人が撮影した写真

## 8. パネル公開方式

パネル公開方式は、ゲーム開始時に以下の2種類から選択できるようにする。

### 方式A：GM承認後に公開

プレイヤーが取得申請を行い、GMが承認した時点で対象パネルの取得処理を行う。

その後、取得したパネルの周囲8マスを公開する。

#### 特徴

- 公平性が高い
- 誤申請による情報公開を防げる
- GMの承認が遅れるとゲーム進行も遅くなる

### 方式B：申請時に仮公開

プレイヤーが取得申請を送信した時点で、対象パネルを審査中とし、その周囲8マスを公開する。

仮公開されたパネルは、その後GMが申請を却下した場合でも公開状態のままとする。

また、仮公開されたパネルにも他プレイヤーは取得申請を行うことができる。

#### 特徴

- ゲームの進行が早くなる
- GMの即時対応が不要になる
- 誤申請・不正申請による情報公開リスクがある
- リスクへの対策としてペナルティポイントを導入する

---

## 9. ペナルティルール

申請時に仮公開する方式を選択した場合、誤申請や不正申請への対策としてペナルティポイントを使用する。

### ペナルティ付与

GMは申請を却下する際、対象チームにペナルティポイントを付与できる。

初期実装では、以下の2操作を用意する。

- 却下する
- 却下してペナルティを付与する

将来的には却下理由ごとにペナルティの有無やポイント数を調整できるようにする。

### ペナルティ発動

ペナルティポイントがゲーム開始時に設定した閾値に達した場合、そのチームは所有しているパネルをランダムに1枚失う。

#### 設定例

```txt
ペナルティ閾値：2ポイント
```

この場合、2ポイントたまるとランダムに1枚パネルを失う。

### パネルを失った場合

失われたパネルは以下の状態になる。

- 所有者なし
- 公開済み
- 破壊状態ではない

つまり、ポケモン情報は公開されたままだが、どのチームの所有でもないパネルとなる。

### ペナルティ発動後

ペナルティによってパネルを失った後、そのチームのペナルティポイントは0に戻る。

## 10. ボーナスルール

特定条件を満たすポケモンを取得した場合、ボーナス効果を発生させる。

### 想定するボーナス

- 色違い
- 実寸大グッズ
- 特定イベント限定
- 推しポケモン

### 推しポケモン

ゲーム開始時に、各プレイヤーまたは各チームが推しポケモンを設定できる。

初期案では、最大3匹までを想定する。

#### 推しポケ取得時の効果案

- 本人または対象チームが取得した場合：周囲3マスを追加取得または追加公開
- 本人以外が取得した場合：周囲2マスを追加取得または追加公開

ただし、ボーナスルールはゲームバランスへの影響が大きいため、初期実装では後回しとする。

---

## 11. アタックチャンス

GMが任意のタイミングでアタックチャンスを発動できる。

アタックチャンスでは、特定のポケモン名を指定するのではなく、条件付きのお題を提示する。

### お題例

- 重たいポケモン
- 小さいポケモン
- 図鑑番号が大きいポケモン
- 高さが一定以上のポケモン
- 名前に特定の文字を含むポケモン

期限内に条件を満たす証拠を提出したプレイヤーは、任意のパネルを1枚所有者なしの状態にできる。

---

## 12. ゲーム終了条件

以下のいずれかを満たした場合、ゲームを終了する。

- 設定した期限になった
- すべてのマスが埋まった
- GMが終了を宣言した
- 不測の事態によりGMが終了判断を行った

GMはゲーム中画面から任意のタイミングでゲーム終了を実行できる。

---

## 13. 勝敗判定

ゲーム終了時に、所有パネル数が最も多いチームを勝利チームとする。

### 同点の場合

所有パネル数が同じ場合は、そのチームが所有するポケモンの中で最も図鑑番号が大きいものを比較する。

より大きい図鑑番号のポケモンを所有しているチームを上位とする。

それでも同じ場合は同率とする。

## 19. 主要データ構造案

### GameSettings

```ts
type GameSettings = {
  boardSize: number;
  timeLimitMinutes: number;
  seed: string;
  pokemonNumberStart: number;
  pokemonNumberEnd: number;
  excludedPokemonNumbers: number[];
  revealMode: "afterApproval" | "onRequest";

```

```ts
type Team = {
  id: string;
  name: string;
  color: string;
  icon?: string;
  players: Player[];
  penaltyPoints: number;
};
```

### Player

```ts
type Player = {
```

## 21. まとめ

初期実装では、まず以下のコアループを完成させる。

```txt
ゲーム設定
↓
ランダム盤面生成
↓
パネル公開
↓
Discordへ証拠画像投稿
↓
システム上で取得申請
↓
GM承認
↓
パネル取得
↓
周囲公開
↓
挟み反転
↓
スコア更新
↓
ゲーム終了
↓
順位表示
```

最初から完全自動化を目指すのではなく、GM操作を前提としたローカル完結版を作成し、ゲームとして成立するかを検証する。

その後、申請機能、仮公開、ペナルティ、リアルタイム同期、外部通知、画像提出などを段階的に追加していく。
